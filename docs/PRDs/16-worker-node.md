## Problem / Context

The user wants to launch coding agents from the outline. "Using Pi as the backend by
default, but I want to be able to provide other agents that support CLI in the future
— so an interface (or Go's equivalent)." Reference: work2/pchain pi-async-subagents
using the Pi extensions API in RPC form. The final UX target: "one line like `○ →
how do I run the tests?  done 44s ↑5.0k ↓2.5k $0.0015` with a one-line summary
generated by a different summary agent while the agent works."

## Goals

- A worker node whose text is a task for a coding agent (Pi by default).
- alt+r runs a turn using the worker node's own text as the task; a second alt+r cancels. (A children-`Context()` concatenation was considered but is not implemented.)
- The worker WORKS (uses tools), does not just chat, and produces no outline nodes — the diff/actions/summary live in an alt+e details page.
- The line stays one line: `┊ model · ↑↓tokens · $`, with a live one-line activity summary while running.
- Provider abstraction so other CLI agents can be added later (planned, not yet built).

## Non-goals

- Appending the agent's answer as a typed child node (tried, then cut).
- An inline scrollable transcript of every tool call (cut).
- A `→` prefix on the worker (reserved for compute nodes).

## Approach / Design

- `pkg/tui/editor/worker.go`: `runWorker` sends the worker node's own text (`it.name`) as the task and calls `startWorker`, which invokes `pi` directly via `exec.CommandContext` with `--mode rpc --no-session --approve --no-extensions` plus lflow's own `finish_worker` tool extension and a system prompt instructing the agent to do the work with tools, then call `finish_worker` once with the deliverable in markdown.
- `startWorker` sends the task as an RPC `prompt`, translates pi's event stream: tool starts become one-line `→ tool args` activity lines; `finish_worker` emits the markdown deliverable; `message_end` accumulates token/cost usage; `agent_end` finishes.
- `workerSuffix` renders the cost chip: `┊ model · ↑Ntok ↓Ntok $0.0000`. Status bar shows the pi model/thinking (`b3a7ccb`).
- Provider abstraction (planned, not implemented): an `AgentProvider` Go interface with a Pi RPC adapter so other CLI agents could plug in later. The shipped code has no such interface; it calls `pi` directly via exec.
- Registered in `registry.go` with no sign, `inlineEditable: true`, `run: runWorker`.

## Decisions

- The worker stays one line and produces no outline nodes; details live on the alt+e page.
- Tool-call traces become a summary-agent one-liner; the deliverable is a single `finish_worker` markdown.
- No `→` icon — `→` is reserved for compute nodes; the worker has no prefix.
- Provider abstraction via an `AgentProvider` interface (Pi adapter) is planned, not shipped — pi is invoked directly via exec for now; keys come from local config (`~/.pi/agent/settings.json`) and never go in argv/history/DB/logs.
- Never auto-run; alt+r only.
- Workers are the basis for comparing models by running the same prompt twice.

## UX / Behavior

- Worker node text is the task; alt+r runs, second alt+r cancels.
- While running: a live one-line activity summary; the chip updates with running tokens/cost.
- Done: one line `┊ model · ↑Ntok ↓Ntok $0.0000`.
- alt+e opens the details page (diff, actions, summary).

### Pivot

Big reversal on the worker UX. Originally the worker was to be exactly like bash —
message + scrollable transcript of every tool call inline, and the agent's final
message appended as a typed child node (so an agent could answer with a bash node or a
structured list). The user cut all of that: "remove agent preview, make agents
respond as single nodes and only show costs," then "no, I asked for it to be one
line." Tool traces became the one-liner summary; the deliverable became a single
`finish_worker` markdown on the alt+e page. The icon changed too: the `→` was removed
and reserved for compute nodes.
